# Paw 모바일 UI/UX 디자인 의뢰서

작성일: 2026-03-14  
대상: UI/UX 디자이너  
목적: Paw의 모바일 네이티브 앱(iOS/Android) 디자인 시스템, 정보구조, 핵심 플로우, 주요 화면 설계를 진행하기 위한 제품 브리프

---

## 1. 서비스 한 줄 정의

**Paw는 사람과 AI 에이전트가 같은 대화 공간에서 실시간으로 협업할 수 있는 AI-네이티브 메신저입니다.**

단순 채팅 앱이 아니라,
- 사람 ↔ 사람 메시징
- 사람 ↔ AI Agent 협업
- 실시간 스트리밍 응답
- 보안 대화(E2EE)
- 그룹/채널/마켓플레이스/권한 관리
를 하나의 메신저 경험으로 통합하는 서비스입니다.

---

## 2. 서비스 목적과 핵심 가치

### 2.1 Paw가 해결하려는 문제
기존 메신저는 사람 간 대화에는 최적화되어 있지만, AI를 같은 수준의 참여자로 다루기에는 구조가 부족합니다. Paw는 다음을 해결하려고 합니다.

1. **사람과 AI가 같은 대화 맥락 안에서 협업**할 수 있게 한다.
2. AI 응답을 단순 봇 메시지가 아니라 **스트리밍, 툴 실행, 카드형 결과, 액션 버튼**까지 포함한 풍부한 메시지로 제공한다.
3. 모바일 환경에서 중요한 **빠른 응답성, 보안, 푸시, 세션 복구, 디바이스 인식**을 기본 전제로 둔다.
4. 장기적으로는 **개인 대화 / 그룹 / 채널 / Agent Marketplace / 보안 / 백업**까지 하나의 제품 경험으로 통합한다.

### 2.2 제품의 차별점
- **AI-native messenger**: AI가 부가 기능이 아니라 대화 구조 안의 핵심 참여자
- **Secure messaging**: OTP + 디바이스 등록 + 향후 E2EE 강화
- **Agent-first UX**: 에이전트 초대, 동의, 스트리밍 응답, 툴 실행 상태 표시
- **실시간 중심 구조**: WebSocket 기반, 대화 흐름이 즉시 반영됨
- **확장성**: 그룹, 채널, 마켓플레이스, 백업, 신고/차단, 관리자 기능까지 확장 가능

## 3. 현재 제품 상태 요약

현재 저장소는 **Flutter 단일 클라이언트에서 모바일 네이티브 앱으로 전환 중**입니다.

### 3.1 현재 클라이언트 구조
- `paw-client/`: Flutter 기반 Web/Desktop 참조 클라이언트
- `paw-ios/`: SwiftUI 기반 iOS 네이티브 쉘
- `paw-android/`: Kotlin/Compose 기반 Android 네이티브 쉘
- `paw-core/`: 모바일 공용 Rust 런타임
- `paw-server/`: Rust/Axum 서버

### 3.2 현재 상태에 대한 해석
- **Flutter 앱은 구조와 UX 참고용 레퍼런스가 가장 풍부함**
- **iOS/Android 네이티브 앱은 foundation shell 수준**으로, auth/bootstrap/chat의 최소 흐름만 존재함
- 즉, 이번 디자인은 **Flutter 레퍼런스를 참고하되, 모바일 네이티브 제품으로 재설계**하는 것이 목적임

## 4. 디자이너가 이해해야 할 제품 특징

### 4.1 핵심 도메인
1. **인증(Auth)**
   - 전화번호 OTP 로그인
   - 디바이스 등록
   - 공개 아이디(username) 설정
   - 세션 복구

2. **대화(Chat / Conversation)**
   - 1:1 대화
   - 그룹 대화
   - 메시지 목록 / 상세
   - 읽음 / 입력 중 / 온라인 상태
   - 첨부 / 미디어 / 검색

3. **보안(Security)**
   - E2EE 활성화/상태
   - 디바이스 키
   - 세션 관리
   - 차단 / 신고 / 권한 동의

4. **AI Agent 경험**
   - Agent가 대화에 참여
   - 대화별 동의(consent)
   - 스트리밍 응답
   - 툴 실행 상태 노출
   - 향후 Agent 상세 / 설치 / 관리 / 마켓플레이스

5. **확장 기능**
   - 채널(Channel)
   - 마켓플레이스(Agent install/discovery)
   - 백업/복원
   - 푸시 알림
   - 운영/안전 기능(신고, 차단, 제재)

## 5. Flutter를 기준으로 파악한 현재 앱 구조

Flutter 라우터 기준 정보구조는 아래와 같습니다.

### 5.1 인증 플로우
- `/login`
- `/auth/phone`
- `/auth/otp`
- `/auth/device-name`
- `/auth/username-setup`

즉 현재 설계 의도는:
1. 로그인 방식 선택
2. 전화번호 입력
3. OTP 입력
4. 기기 이름 설정
5. 공개 아이디(username) 설정
6. 메인 진입

### 5.2 메인 쉘 구조
메인 내비게이션은 3축입니다.
- `/chat`
- `/agent`
- `/settings`

Flutter 기준:
- 모바일: 하단 내비게이션
- 데스크톱: 좌측 사이드바

모바일 네이티브 디자인에서도 이 3축은 중요한 출발점이 됩니다.

### 5.3 채팅 구조
- 대화 목록 `ConversationsScreen`
- 대화 상세 `ChatScreen`
- 대화 검색 `SearchScreen`
- 그룹 만들기 `CreateGroupScreen`
- 그룹 정보 `GroupInfoScreen`
- 키 검증 `KeyVerificationScreen`

현재 대화 화면에서 드러나는 정보 요소:
- 대화명 / 아바타
- Agent 참여 여부
- E2EE 여부
- unread badge
- 마지막 메시지 미리보기
- 타임스탬프
- 대화 필터(전체 / 보안 / Agent / 안읽음)

### 5.4 채팅 상세 구조
채팅 상세 화면은 현재 다음 레이어를 가짐:
- 상단 헤더(대화명, 상태, 보안/Agent 여부)
- Agent consent banner
- E2EE banner
- WebSocket 상태 배너
- 메시지 리스트
- 스트리밍 AI 버블
- 타이핑 인디케이터
- 메시지 입력창

메시지 타입은 다음까지 고려되어 있습니다.
- 일반 텍스트
- 마크다운
- 미디어 메시지
- AI 스트리밍 메시지
- 툴 실행 표시
- 카드 블록 / 액션 버튼 블록

### 5.5 Agent 영역 구조
현재 Flutter 상 Agent 탭은 다음 역할을 암시합니다.
- 추천 Agent 노출
- 카테고리 탐색
- Agent 검색
- 설치 여부 확인
- Agent 열기 / 추가
- 권한 가이드 진입

즉 장기적으로는 **메신저 안의 앱스토어/도구 허브** 성격입니다.

### 5.6 설정/프로필 구조
- Settings
- My Profile
- User Profile
- Help Center

현재 의도된 설정 영역:
- 알림
- 개인정보(읽음 표시, 입력중 표시, 언어)
- 보안(세션, 보안 점검, 다크모드)
- AI/Agent 권한
- 도움말
- 로그아웃

## 8. 아직 UI/UX에 충분히 담기지 못한 기능들

아래는 현재 코드/화면상 미완성 또는 stub 상태로 보이는 항목입니다.

### 8.1 인증/계정
- username 로그인은 아직 준비 중
- 디바이스 등록 UX가 기능적으로만 존재하고 제품화되지 않음
- 세션/기기 관리 화면 부재

### 8.2 채팅
- 미디어 선택/업로드 UX 미완성
- 이미지/파일 상세 뷰어 미완성
- 그룹 생성 시 멤버 선택 UX 부재
- 그룹 이름 수정 / 멤버 추가 / 그룹 나가기 미완성
- 메시지 반응 / 스레드 UX 부재
- 음성/영상 확장 고려 필요

### 8.3 Agent
- Agent 상세 화면 부재
- Agent 설치/열기 플로우 미완성
- 대화 안에서 Agent 권한 관리 UX 미완성
- 스트리밍/툴 실행/카드 응답 디자인 체계 미정교화

### 8.4 보안
- E2EE 활성화/키 검증 UX가 개념 수준
- 웹에서의 제한 안내는 있으나, 네이티브에서의 보안 UX 정교화 필요
- 차단/신고/콘텐츠 안전 플로우 부재

### 8.5 플랫폼/운영
- 채널 UI 없음
- 백업/복원 UI 없음
- 푸시 설정과 알림 센터 구조 미흡
- 오프라인/재연결/에러 상태 설계 부족
- 빈 상태 / 실패 상태 / 권한 거부 상태 / 만료 상태 정교화 필요

## 10. 비주얼 톤 & 스타일 가이드의 출발점

현재 코드에서 드러나는 시각적 방향은 다음과 같습니다.

### 10.1 톤
- **Dark-first**
- **Warm editorial palette**
- 차가운 블랙이 아니라 **짙은 올리브/브라운 계열의 다크 톤**
- 포인트 컬러는 **amber/copper 계열 accent**

### 10.2 형태
- 과도하게 둥글지 않은 **낮은~중간 radius**
- 박스형 surface 분리
- 카드/패널/배너가 명확히 층위를 형성

### 10.3 텍스트 톤
현재 레퍼런스는 모노스페이스 중심으로 설계돼 있으나, 실제 모바일 제품에서는 재검토가 필요합니다.

특히 한국어 UX 관점에서 권장:
- **한글 본문에 과도한 모노스페이스 사용은 지양**
- 코드/OTP/메타 정보만 모노 계열 사용
- 본문/탐색/설명은 가독성 높은 일반 UI 텍스트 체계 사용

즉, 디자이너는 **현재의 'editorial / technical / premium' 무드**는 유지하되,
실사용성을 위해 **한글 중심 가독성**을 강화하는 방향을 제안해 주시면 좋습니다.

## 11. 모바일 UX 설계 시 반드시 고려할 제약

### 11.1 모바일 네이티브 관점
- 한 손 사용성
- 빠른 대화 전환
- 긴 메시지와 스트리밍 메시지 가독성
- 키보드/입력창/첨부 UX
- 푸시 알림과 앱 복귀 흐름
- 오프라인 / 재접속 / 복구 시나리오

### 11.2 보안 관점
- 인증 상태를 명확히 전달할 것
- Agent 접근 권한은 대화별로 분리해서 이해 가능해야 함
- E2EE가 켜진 대화와 그렇지 않은 대화를 헷갈리지 않게 해야 함
- 디바이스/세션 관련 정보는 위협감을 주지 않으면서도 신뢰감을 줘야 함

### 11.3 AI UX 관점
- AI 응답은 일반 사용자 메시지와 구분돼야 함
- 하지만 “봇 같음”보다 “협업 파트너”처럼 느껴지도록 설계 필요
- 스트리밍 중 상태, 툴 실행 중 상태, 완료 상태의 차이를 명확히 표현할 것
- 카드/추천/액션 버튼은 대화 맥락을 끊지 않도록 설계할 것
